# AI 员工系统修改指导文档

本文档用于交给其他智能体或开发者继续修改 `ai-employee-prototype-app`。目标不是继续堆页面，而是把当前原型升级为更像真实 AI 员工平台的产品：能接任务、会规划、可执行、可审计、能衡量效果。

## 1. 当前判断

当前项目已经具备较完整的原型页面和业务模块：

- 内容搜索
- AI 视频生成
- 客户消息
- 平台账号与抖音授权
- 商户入驻
- 商品选品
- 达人合作
- 内容发布
- 订单库存
- 财务结算
- 团队权限
- 经营分析

但系统仍主要是原型：

- `src/store.js` 使用内存数据，重启后状态丢失。
- 演示账号密码明文写在代码里。
- 多平台内容、客服、订单、销售等数据大量为模拟数据。
- AI 能力以脚本生成、视频生成、规则回复为主，还没有形成“任务规划 -> 工具调用 -> 人工审批 -> 执行记录”的 AI 员工闭环。
- 前端 `public/app.js` 文件过大，页面逻辑、事件绑定、渲染模板集中在一个文件中，后续维护风险高。
- 界面已有业务感，但还不够“科技化、智能化”，缺少 AI 工作状态、执行轨迹、智能推荐、实时感和可视化中心。

## 2. 总体修改目标

把系统从“多功能演示工作台”升级为“AI 电商运营员工平台”。

核心目标：

1. 用户能给 AI 员工下达业务目标。
2. AI 员工能拆解任务、调用工具、生成结果。
3. 高风险动作需要人工审批。
4. 所有 AI 决策、工具调用、平台同步和成本都可追踪。
5. 界面呈现更强的科技感、智能感、实时感和可信度。

## 3. 推荐分阶段实施

### 阶段一：生产底座补强

目标：让系统具备可持续开发和真实试用的基础。

建议任务：

- 增加数据库层，优先使用 SQLite 或 PostgreSQL。
- 将 `store.js` 内存数据迁移为 repository/service 模式。
- 增加用户表、租户表、角色表、权限表、额度表。
- 替换明文密码，使用密码哈希。
- 增加真实 session 或 JWT 登录机制。
- 增加 API 输入校验。
- 增加统一错误响应格式。
- 增加基础测试。
- 补齐缺失的 `README.md`。

建议修改范围：

- `src/store.js`
- `src/server.js`
- `src/modules/onboarding.js`
- 新增 `src/db/`
- 新增 `src/repositories/`
- 新增 `src/middleware/`
- 新增 `README.md`

验收标准：

- 重启服务后用户、任务、视频草稿、会话数据不丢失。
- 不能再使用明文密码登录逻辑。
- `npm run check` 通过。
- 至少覆盖登录、用户额度、视频草稿、客户会话的基础测试。

### 阶段二：AI 员工任务中心

目标：让产品从“工具集合”变成“AI 员工系统”。

新增核心模块：任务中心。

任务中心应支持：

- 创建 AI 任务。
- 选择任务类型：内容研究、视频生成、客服回复、发布计划、数据分析、订单售后。
- AI 自动拆解子任务。
- 展示任务状态：等待、规划中、执行中、待审批、已完成、失败。
- 展示执行时间线。
- 展示 AI 调用过的工具。
- 展示模型成本和消耗额度。
- 支持人工批准、驳回、重试、取消。
- 支持任务结果沉淀为案例或模板。

建议新增后端模型：

- `tasks`
- `task_steps`
- `agent_runs`
- `tool_calls`
- `approval_requests`
- `cost_records`

建议新增页面：

- `任务中心`
- `AI 员工控制台`
- `执行日志`
- `审批中心`

建议修改范围：

- `src/server.js`
- 新增 `src/modules/tasks.js`
- 新增 `src/modules/agents.js`
- 新增 `src/modules/approvals.js`
- `public/index.html`
- `public/app.js` 或拆分后的前端模块
- `public/styles.css`

验收标准：

- 用户可以创建一个“生成短视频并准备发布计划”的任务。
- 系统能拆成至少 3 个步骤：分析商品、生成脚本分镜、生成发布建议。
- 高风险操作如“发布到平台”“自动回复客户”“修改订单状态”必须进入审批。
- 页面能展示 AI 执行时间线和每一步结果。

### 阶段三：真实平台闭环

目标：优先打通一个真实平台，不要同时浅接多个平台。

推荐优先平台：抖音。

建议任务：

- 完善抖音授权状态、token 刷新、错误提示。
- 将内容搜索结果、账号信息、平台消息同步持久化。
- 为平台 API 调用增加限流、重试、失败记录。
- 增加平台能力矩阵，明确每个平台支持哪些能力。
- 将当前 `local_gateway_queue` 升级为真实消息队列或待发送记录。

建议平台能力矩阵字段：

- 内容搜索
- 评论读取
- 私信读取
- 评论回复
- 私信回复
- 视频发布
- 商品同步
- 订单读取
- 售后读取

建议修改范围：

- `src/modules/douyin.js`
- `src/modules/content.js`
- `src/modules/messages.js`
- `src/modules/publishing.js`
- 新增 `src/platforms/`

验收标准：

- 页面能清晰显示真实接口、模拟数据、审核中、未授权四种状态。
- 抖音真实接口失败时，错误被记录到任务日志，不只显示在页面状态栏。
- 平台同步动作都有唯一记录 ID。

### 阶段四：知识库与智能客服升级

目标：把客服从“话术生成”升级为可控的销售/售后 AI。

建议任务：

- 将单个知识库文本框升级为结构化知识库。
- 支持商品资料、FAQ、售后政策、物流政策、价格策略、活动规则。
- 引入检索增强生成，即 RAG。
- 客服回复必须引用命中的知识来源。
- 增加升级人工规则。
- 增加敏感场景识别：投诉、退款、差评、质量问题、法律风险。
- 增加回复前审批选项。
- 增加客服质量评分。

建议修改范围：

- `src/modules/messages.js`
- `src/modules/settings.js`
- 新增 `src/modules/knowledge.js`
- 新增 `src/services/retrieval-service.js`
- 前端新增知识库管理页面

验收标准：

- 客服回复不再只拼接固定模板。
- 每条 AI 回复能显示依据来源。
- 命中投诉、退款、质量问题时，自动转为人工处理或审批。
- 可查看每条回复的命中知识、模型、生成时间、风险标签。

### 阶段五：AI 视频能力产品化

目标：把 AI 视频从单次生成工具升级为内容生产流水线。

建议任务：

- 增加素材库：商品图、场景图、品牌图、历史视频、口播音色。
- 增加品牌规范：主色、禁用词、固定片尾、Logo、水印策略。
- 增加视频批量生成。
- 增加分镜版本管理。
- 增加不同平台尺寸：9:16、1:1、16:9。
- 增加模型能力矩阵和供应商状态。
- 增加异步任务队列，避免视频生成阻塞 HTTP 请求。
- 增加失败重试、取消任务、生成成本统计。
- 增加版权和合规检查。

建议修改范围：

- `src/modules/video.js`
- `src/services/ai-video-service.js`
- `src/services/video-renderer.js`
- `src/modules/settings.js`
- 新增 `src/modules/assets.js`
- 新增 `src/modules/brand.js`

验收标准：

- 用户可以从素材库选择商品图生成视频。
- 视频生成过程在任务中心显示，不阻塞页面。
- 每个视频结果记录模型、供应商、成本、耗时、素材来源。
- 不支持的供应商不能只显示在下拉框中，必须明确标记为“未接入”或“配置中”。

### 阶段六：经营分析与 ROI 闭环

目标：让商户知道 AI 员工到底有没有产生业务价值。

建议任务：

- 建立内容 -> 发布 -> 流量 -> 客服 -> 成交 -> 售后 -> 复购的转化链路。
- 增加 AI 任务 ROI 统计。
- 增加渠道、商品、视频、客服话术维度分析。
- 增加 A/B 测试。
- 增加经营建议自动生成。
- 增加异常预警：转化率下降、库存不足、退款上升、差评集中。

建议修改范围：

- `src/modules/analytics.js`
- `src/modules/sales.js`
- `src/modules/orders.js`
- `src/modules/finance.js`
- 新增 `src/modules/experiments.js`
- 新增经营分析页面组件

验收标准：

- 每个 AI 任务能关联业务结果。
- 能看出哪条视频、哪类客服回复、哪个商品带来更高转化。
- AI 建议不能只显示静态文本，要能基于真实指标生成。

## 4. 前端架构修改建议

当前 `public/app.js` 过大，建议逐步拆分。

推荐目录结构：

```text
public/
  app.js
  styles.css
  index.html
  js/
    api.js
    state.js
    router.js
    utils.js
    components/
      modal.js
      stat-card.js
      data-table.js
      agent-timeline.js
      task-status.js
    pages/
      dashboard.js
      task-center.js
      ai-console.js
      content.js
      video.js
      messages.js
      publishing.js
      products.js
      creators.js
      orders.js
      finance.js
      team.js
      settings.js
```

拆分原则：

- `api.js` 只负责请求。
- `state.js` 只维护前端状态。
- `router.js` 只处理页面切换。
- `components/` 放通用 UI。
- `pages/` 放业务页面。
- 禁止继续把新页面追加到单个 `app.js` 末尾。

验收标准：

- 新增页面不需要修改超过 3 个无关模块。
- 页面渲染函数可单独阅读。
- 公共组件不复制粘贴。

## 5. 界面整体改造方向

界面目标：科技化、智能化、可信赖、适合高频运营。

不要做成纯展示型官网，也不要做成花哨大屏。它应该像一个真实 AI 工作系统：信息密度高、状态清晰、动作可控。

### 5.1 视觉关键词

- 智能指挥中心
- 实时任务流
- 数据驾驶舱
- AI 执行轨迹
- 人机协同
- 可控自动化
- 商业可信赖

### 5.2 推荐布局

保留左侧导航，但首页改成 AI 控制台。

首页建议区域：

1. 顶部 AI 员工状态栏
   - 当前在线 agent 数量
   - 今日执行任务数
   - 待审批数量
   - 今日节省时间
   - 今日模型成本

2. 中央任务指挥区
   - 输入业务目标
   - 快捷任务模板
   - AI 自动拆解预览
   - 一键创建任务

3. 右侧实时动态栏
   - AI 正在分析商品
   - AI 正在生成分镜
   - AI 等待你审批客服回复
   - 平台同步失败提醒

4. 下方经营智能洞察
   - 今日异常
   - 增长机会
   - 推荐动作
   - 风险提示

### 5.3 新增核心页面

#### AI 员工控制台

用途：用户给 AI 下达目标。

界面元素：

- 目标输入框：例如“为防护手套生成 10 条抖音短视频并制定发布计划”
- 任务模板：内容生产、客服转化、订单售后、经营分析
- Agent 选择器：内容研究员、视频编导、客服销售、数据分析师
- 任务预估：预计耗时、预计成本、需要审批的步骤
- 创建任务按钮

#### 任务中心

用途：展示所有 AI 工作。

界面元素：

- 状态筛选：全部、执行中、待审批、失败、完成
- 任务列表
- 任务详情抽屉
- 执行时间线
- 工具调用记录
- 成本记录
- 审批操作

#### 审批中心

用途：处理高风险自动化动作。

需要审批的动作：

- 自动回复客户
- 发布内容到平台
- 修改订单状态
- 发送优惠券
- 修改商品价格
- 导出客户数据

界面元素：

- 风险等级
- AI 建议内容
- 命中知识依据
- 影响对象
- 批准、驳回、编辑后批准

#### AI 观察日志

用途：提高可信度。

界面元素：

- Agent 名称
- 输入
- 思考摘要
- 工具调用
- 输出
- 失败原因
- 耗时
- 成本
- 人工干预记录

## 6. 科技化界面风格规范

### 6.1 色彩建议

推荐使用深浅混合的专业科技风，不要全页面单一蓝紫。

主背景：

- 深色模式：`#0B1020`
- 页面底色：`#0F172A`
- 面板底色：`#111827`
- 浅色模式备用：`#F7F9FC`

强调色：

- 智能蓝：`#2F80ED`
- 电光青：`#14B8A6`
- 数据绿：`#22C55E`
- 风险橙：`#F59E0B`
- 错误红：`#EF4444`

中性色：

- 主文字：`#E5E7EB`
- 次文字：`#94A3B8`
- 边框：`#263244`
- 分割线：`#1E293B`

要求：

- 不要整站只用蓝紫渐变。
- 不要大面积使用纯黑。
- 不要使用过多发光阴影。
- 核心状态色必须有业务含义。

### 6.2 字体和密度

建议：

- 中文字体使用系统字体栈。
- 标题克制，不要做营销页大标题。
- 工作台字号以 13px - 16px 为主。
- 数据数字可使用 24px - 32px。
- 任务状态、日志、表格要适合扫描。

要求：

- 不要使用视口宽度控制字体大小。
- 不要负字距。
- 按钮文字必须在移动端完整显示。

### 6.3 组件风格

推荐组件：

- 状态芯片
- 时间线
- 任务卡片
- 数据表格
- 右侧详情抽屉
- 命令输入框
- 审批卡片
- Agent 头像标识
- 模型成本标签
- 工具调用徽标

圆角：

- 普通卡片 8px 以内。
- 按钮 6px - 8px。
- 不要过度圆润。

图标：

- 建议引入 `lucide` 或同类线性图标库。
- 当前 emoji 图标可以逐步替换为专业图标。
- 图标按钮需要 tooltip。

动画：

- 任务执行可用轻量脉冲、进度条、扫描线。
- 审批和错误状态不要使用夸张动画。
- 动画时长建议 120ms - 240ms。
- 不要用大面积漂浮光球、渐变 blob、纯装饰粒子。

### 6.4 智能化表达方式

界面不要只显示“AI 已生成”，要显示“AI 为什么这么做”。

每个 AI 结果建议显示：

- 使用的模型
- 使用的知识来源
- 置信度或风险等级
- 关联任务
- 生成时间
- 可执行下一步

示例：

```text
AI 建议回复
依据：商品 FAQ / 售后政策 / 客户历史咨询
风险：低
下一步：发送 / 编辑 / 转人工
```

### 6.5 首页改造草图

建议首页信息结构：

```text
+----------------------------------------------------------+
| AI 员工系统 | 当前 4 个 Agent 在线 | 待审批 3 | 成本 ¥18.6 |
+----------------------+----------------------+------------+
| 左侧导航             | 业务目标输入区       | 实时动态   |
| - AI 控制台          | 输入目标             | 任务流     |
| - 任务中心           | 任务模板             | 风险提醒   |
| - 审批中心           | 预估成本/耗时        | 平台状态   |
| - 内容运营           | 创建任务             |            |
| - 客户管理           +----------------------+------------+
| - 经营分析           | 今日智能洞察                       |
| - 设置               | 异常 / 机会 / 推荐动作 / ROI       |
+----------------------+-----------------------------------+
```

## 8. 建议分配给智能体的任务包

### 智能体 A：后端底座

范围：

- 数据库
- 登录
- 用户权限
- API 校验
- repository 层

禁止修改：

- UI 视觉样式
- 视频生成算法

交付：

- 数据库初始化脚本
- 登录与用户模块改造
- 数据持久化验证说明

### 智能体 B：任务中心与 Agent 编排

范围：

- 任务模型
- agent run
- tool call
- approval
- 任务 API

禁止修改：

- 平台 API 细节
- 财务计算逻辑

交付：

- 任务中心后端 API
- 示例任务流程
- 审批流程

### 智能体 C：前端架构与页面

范围：

- 拆分 `public/app.js`
- 新增 AI 控制台
- 新增任务中心
- 新增审批中心
- 新增执行日志组件

禁止修改：

- 后端业务规则
- 数据库 schema

交付：

- 新页面可运行
- 移动端布局可用

## 9. 全局验收清单

开发完成后必须检查：

- `npm run check` 通过。
- 主要页面可打开。
- 登录、退出、切换账号可用。
- 任务中心可创建任务。
- AI 任务有执行时间线。
- 待审批动作不会自动执行。
- 客服回复能显示依据。
- 视频生成不阻塞页面请求。
- 真实数据和模拟数据有明确标识。
- 页面在 1366px、1920px、390px 宽度下不重叠。
- 按钮文字不溢出。
- 颜色不形成单一蓝紫主题。
- 新增代码不继续扩大单体 `app.js`。
- 文档更新同步完成。

## 10. 给后续智能体的原则

- 不要只新增页面，要补业务闭环。
- 不要把模拟数据伪装成真实数据。
- 不要让 AI 自动执行高风险动作。
- 不要继续把所有前端逻辑写入一个文件。
- 不要用过度装饰替代清晰状态。
- 每个 AI 结果都要能解释来源、成本、风险和下一步。

最终产品应该让用户感觉：这不是一个“AI 生成工具合集”，而是一个正在替我工作的 AI 运营团队。
